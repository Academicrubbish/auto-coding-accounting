# 自动化 Debug 工作流设计

## 设计理念

基于 Anthropic 的《Effective Harnesses for Long-Running Agents》文章的核心思想：

### 核心问题
1. **上下文丢失** - 每次新的 Agent 会话没有之前工作的记忆
2. **过早声明完成** - Agent 倾向于一次性做太多，导致功能未完成
3. **缺少验证** - 代码变更后没有充分测试就标记完成
4. **状态混乱** - 每个 session 结束后留下未完成的代码

### 解决方案
1. **Initializer Agent** - 设置初始环境（功能清单、进度追踪、初始化脚本）
2. **Debug Agent** - 每次 session 专注于修复一个 bug，留下清晰的 artifacts
3. **Feature List** - 结构化的测试场景定义（JSON 格式）
4. **Testing** - 强制进行端到端测试，使用浏览器自动化工具
5. **Clean State** - 每个 session 结束时要 git commit 并记录进度

## 核心组件设计

### 1. Initializer Agent（初始化 Agent）

**职责**：首次运行时设置所有必要的环境和 artifacts

**输出 artifacts**：

#### a) `task.json` - 测试场景清单（不可变）
```json
{
  "version": "1.0.0",
  "project": "accounting-app",
  "generated_at": "2026-03-09T10:00:00Z",
  "scenarios": [
    {
      "id": "auth-001",
      "name": "微信登录完整流程",
      "category": "user-auth",
      "priority": "critical",
      "description": "用户首次进入触发微信授权登录",
      "steps": [
        "打开小程序/应用",
        "点击微信授权登录按钮",
        "确认授权弹窗",
        "验证登录成功并跳转首页",
        "验证用户信息正确显示（头像、昵称）",
        "验证 openid 已存储到本地"
      ],
      "test_criteria": [
        "登录成功后跳转到首页",
        "用户头像和昵称正确显示",
        "本地缓存存储了 openid",
        "store 中 userData 状态正确更新"
      ],
      "status": "pending",
      "bugs": []
    }
    // ... 更多测试场景
  ]
}
```

#### b) `init.sh` - 环境初始化脚本
```bash
#!/bin/bash
# 记账小程序 - 自动化 Debug 工作流初始化脚本

set -e  # 遇到错误立即退出

PROJECT_DIR="D:/mystudyspace/test/auto-coding-accounting/account-app"

echo "🚀 初始化记账小程序测试环境..."

# 1. 检查项目目录
if [ ! -d "$PROJECT_DIR" ]; then
    echo "❌ 项目目录不存在: $PROJECT_DIR"
    exit 1
fi

cd "$PROJECT_DIR"

# 2. 安装依赖（如果需要）
if [ ! -d "node_modules" ]; then
    echo "📦 安装依赖..."
    npm install
fi

# 3. 启动开发服务器
echo "🔧 启动开发服务器..."
npm run dev:mp-weixin &
DEV_SERVER_PID=$!

# 等待服务器启动
sleep 5

# 4. 运行基础冒烟测试
echo "🔍 运行基础冒烟测试..."
# 这里可以添加基础测试命令

echo "✅ 初始化完成！"
echo "📝 下一步：运行 Debug Agent 开始测试和修复"
```

### 2. Debug Agent（调试 Agent）

**每个 session 的工作流程**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Debug Agent Session 流程                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Step 1: 环境准备                         │   │
│  │  - 运行 pwd 确认工作目录                                      │   │
│  │  - 运行 init.sh 启动测试环境                                  │   │
│  │  - 等待服务启动完成                                           │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Step 2: 了解当前状态                     │   │
│  │  - 读取 progress.txt 了解上次进度                             │   │
│  │  - 读取 task.json 查看测试场景                                │   │
│  │  - 运行 git log 查看最近提交                                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Step 3: 选择测试任务                     │   │
│  │  - 从 task.json 选择一个 pending 状态的场景                   │   │
│  │  - 优先级: critical > high > medium > low                    │   │
│  │  - 考虑依赖关系选择合适的场景                                 │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Step 4: 执行测试                         │   │
│  │  - 使用微信开发者工具 MCP 执行测试步骤                       │   │
│  │  - 逐步验证每个测试点                                        │   │
│  │  - 截图记录测试过程                                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ▼                                       │
│                    ┌─────────────┐                                   │
│                    │ 测试是否通过? │                                   │
│                    └─────────────┘                                   │
│                      /         \                                      │
│                   ✅           ❌                                     │
│                    │             │                                     │
│                    ▼             ▼                                     │
│  ┌─────────────────────────┐  ┌─────────────────────────────────┐    │
│  │  Step 5a: 记录通过       │  │  Step 5b: 分析并修复 Bug        │    │
│  │  - 更新 task.json 状态   │  │  - 分析 bug 原因                │    │
│  │  - 更新 progress.txt     │  │ - 定位问题代码                  │    │
│  │  - Git commit           │  │ - 实现修复方案                  │    │
│  └─────────────────────────┘  │ - 重新测试验证                  │    │
│                              │ - 记录 bug 和修复                │    │
│                              └─────────────────────────────────┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**关键规则**：

1. **一次只修复一个 bug** - 专注于单一问题，避免过度修改
2. **必须端到端测试** - 使用微信开发者工具 MCP 进行真实测试
3. **留下清晰记录** - 更新 progress.txt 和 task.json
4. **保持干净状态** - 每次结束前 git commit

## 工具集成

### 微信开发者工具 MCP

用于自动化测试小程序：

```javascript
// 连接开发者工具
await connect_devtools({
  projectPath: "D:/mystudyspace/test/auto-coding-accounting/account-app",
  strategy: "auto"
})

// 获取页面快照
const snapshot = await get_page_snapshot({ format: "compact" })

// 点击元素
await click({ uid: "button-login" })

// 输入文本
await input_text({ uid: "input-amount", text: "100" })

// 验证状态
await assert_state({ uid: "user-avatar", visible: true })
await assert_text({ uid: "user-nickname", textContains: "用户昵称" })

// 导航
await navigate_to({ url: "/pages/index/index" })

// 断开连接
await disconnect_devtools()
```

## 常见失败模式与解决方案

| 问题 | Initializer Agent | Debug Agent |
|-----|------------------|-------------|
| Agent 声称测试通过但实际有 bug | 创建详细的测试场景和验证标准 | 强制使用浏览器/小程序工具进行端到端测试 |
| Agent 一次修改太多导致混乱 | 设置场景优先级，一次只测一个 | 每次只修复一个 bug，完成后立即 commit |
| Agent 丢失上下文 | 创建清晰的 progress.txt | 开始时读取 progress.txt 和 git log |
| 测试不充分 | 在场景中定义详细的测试步骤 | 使用自动化工具进行真实测试 |
| 环境配置问题 | 创建 init.sh 自动化环境设置 | 每次开始时运行 init.sh |

## 工作流启动指令

### 首次运行（Initializer Agent）

```
你是这个记账小程序自动化 Debug 项目的 Initializer Agent。

你的任务是：
1. 分析项目结构，理解所有功能模块
2. 根据功能模块清单创建完整的测试场景（task.json）
3. 创建 init.sh 脚本用于启动测试环境
4. 创建 progress.txt 进度文件
5. 初始化 git 仓库（如果还没有）

重要：
- task.json 一旦创建就不应再被修改（除了更新 status 字段）
- 场景要详细、具体、可验证
- 每个 bug 都要有独立的 ID 和追踪状态
```

### 后续运行（Debug Agent）

```
你是这个记账小程序的 Debug Agent。

每次会话的工作流程：
1. 运行 init.sh 启动测试环境
2. 读取 progress.txt 和 task.json 了解当前状态
3. 选择一个待测试的场景（优先级高的优先）
4. 使用微信开发者工具 MCP 执行测试
5. 如果发现 bug：
   - 记录 bug 信息到 task.json
   - 分析并修复 bug
   - 重新测试验证
6. 如果测试通过：
   - 更新场景状态为 passed
7. 更新 progress.txt 记录本次工作
8. Git commit 所有更改

重要规则：
- 一次只处理一个场景
- 必须使用真实工具进行端到端测试
- 不要跳过测试步骤
- 每次结束必须留下干净的状态
```
